# Security

## Authentication

Admin access is protected by a JWT stored in an encrypted, HTTP-only cookie.

- **Login URL:** `/superadmin/login` — no link to this page exists on public-facing views.
- **Token:** HS256, signed with `JWT_SECRET` env var (falls back to `Rails.application.secret_key_base`).
- **Cookie:** `admin_token`, encrypted by Rails, HTTP-only, `SameSite: Lax`, `Secure` in production, expires in 12 hours.
- **Guard:** `Admin::BaseController#require_admin!` decodes the token and checks `user.admin? == true` on every admin request. A valid token for a non-admin user is rejected.
- **No session state server-side** — the JWT is self-contained.

## Passwords

Passwords are hashed with Argon2 (`password_digest` column). Plaintext passwords are never stored or logged. The minimum password length is 12 characters.

## Markdown sanitization

Post content is written in Markdown and rendered to HTML by Redcarpet. Before being sent to the browser, the HTML is sanitized by `ActionController::Base.helpers.sanitize` with an explicit allowlist:

**Allowed tags:** `a blockquote br code del em figcaption figure h1–h6 hr img li ol p pre strong table tbody td th thead tr ul`

**Allowed attributes:** `alt href rel src target title`

`<script>` tags and `javascript:` URLs are stripped. All links get `rel="nofollow noopener"` and open in a new tab.

## Strong parameters

All admin controllers use `params.require(:model).permit(...)` with an explicit allowlist. The post controller permits only `body_markdown`, `status`, and `published_at` — title, slug, and excerpt are derived server-side and cannot be submitted directly.

## Image storage

Active Storage is configured with `resolve_model_to_route: :rails_storage_proxy` — image URLs go through Rails, not directly to MinIO. MinIO buckets are private. Direct object access is not possible without a signed URL generated by Rails.

## Reader identity

Reader reactions are tracked by a UUID stored in the Rails session cookie (`reader_id`), set automatically on first visit. This is not a user account — it provides no authentication and cannot be used to access any protected resource. Comment author names and optional emails are stored as plain text and are never used for authentication.

## Secrets

Never commit secrets. In the dev container, credentials are set in `docker-compose.yml` for local-only use. For production, use environment variables or Rails encrypted credentials — never hardcode values in source files.
